# Cairn — start here

This folder is ready to be dropped into an empty repo as the root. It contains:

- `docs/spec.md` — the Cairn v0.6 specification
- `openspec/changes/cairn-kernel-mvp/` — the first implementation change, in OpenSpec format
- `test/fixtures/cairn-bootstrap/` — Cairn described as a Cairn project (self-hosting test fixture)

Everything else (OpenSpec scaffolding, Conflux config, Claude skills) gets generated by the init commands below.

## Prerequisites

You need:

- **Rust + Cargo** — install from https://rust-lang.org/tools/install
- **Claude Code** — install from https://claude.com/product/claude-code
- **Node.js** (for OpenSpec CLI)

Verify:

```bash
cargo --version
claude --version
node --version
```

## Setup

From the terminal, in the directory where you want the Cairn repo:

```bash
# 1. Make the repo and drop these files in as the root
mkdir cairn
cd cairn
git init

# Unpack this bundle at the repo root so you get docs/, openspec/changes/, test/fixtures/
# (If you're reading this file, you've already done this part.)

# 2. Install OpenSpec globally and initialise it in this repo
npm install -g @fission-ai/openspec@latest
openspec init
# When prompted, select Claude Code from the tool list.
# This will notice the existing openspec/changes/ directory and leave it alone.

# 3. Install Conflux
cargo install cflx
cflx --version    # confirm

# 4. Install Conflux's bundled Claude skills
cflx install-skills --claude

# 5. Generate the Conflux config
cflx init

# 6. Commit the scaffolding (Conflux requires a clean worktree before running)
git add -A
git commit -m "Initial setup: spec v0.6, MVP change, OpenSpec + Conflux scaffolding"
```

## Run it

Two options.

### Option A — Conflux TUI (interactive)

```bash
cflx
```

TUI controls:

- `↑/↓` or `j/k` — move between changes
- `Space` — select the `cairn-kernel-mvp` change
- `F5` — start it running
- `Ctrl+C` — exit

### Option B — Conflux headless (runs unattended)

```bash
cflx run --change cairn-kernel-mvp
```

Either way, Conflux drives Claude Code through `openspec/changes/cairn-kernel-mvp/tasks.md`. When it reaches `merged`, phase 1 is done.

## What you should see when it's done

- `src/` — TypeScript source implementing the parser, graph, and CLI
- `bin/cairn.js` — the CLI entry point
- `test/` — tests including the self-hosting test against `test/fixtures/cairn-bootstrap/cairn.dsl`
- `openspec/changes/archive/YYYY-MM-DD-cairn-kernel-mvp/` — the archived change
- `openspec/specs/` — the merged specs promoted from the change

Verify by running:

```bash
node bin/cairn.js get cairn.kernel.parser --file test/fixtures/cairn-bootstrap/cairn.dsl
```

If that returns node metadata, the MVP works.

## Troubleshooting

**"openspec init complains about existing directory."** Should be fine — it handles existing `openspec/changes/` gracefully. If it doesn't, temporarily rename the directory, run `openspec init`, then rename it back.

**"Conflux says the worktree isn't clean."** Commit everything first. Conflux requires a clean git state before running.

**"Claude Code asks clarifying questions mid-run."** Answer using `docs/spec.md` as the source of truth. If the MVP proposal and the spec disagree anywhere, the spec wins.

**"Tests fail on the self-hosting step (task 7)."** That's a signal the parser isn't ready for the full grammar. Iterate on the parser, not the fixture.

**"Build takes longer than expected."** Two focused days is the target. If it stretches past that, stop and re-read the spec. The issue is almost always in the spec, not the code.

## When phase 1 is done

You'll want to either:

- Write a phase 2 change proposal (full artefact type system) and run it the same way, or
- Take a break and run the side-by-side test — use the MVP on a real project (MAG is a good candidate) and see if the structural queries actually improve the coding experience versus OpenSpec alone.

The side-by-side test is what tells you whether Cairn is worth continuing to build.
