## Q2 deferred engine: DoltLite BLOCKED on documented data loss (not adoptable now)

The cross-device merge engine is **not** DoltLite for now. DoltHub's own engineer reviewed the (vibe-coded) DoltLite codebase on 2026-04-08 and reproduced silent data loss under concurrent connections plus a durability bug, exactly the multi-machine scenario it would be used for (details below). Until those are demonstrably fixed, the default and the actual multi-place path is git-merged structured text (JSON/TOML) + content-hash IDs, syncing through the existing git remote (row-level merge for free, zero new engine). Reasoning:

- **The one question:** on offline divergent edits to the same status/claim, do you want auto-resolution to a deterministic winner with the conflict detectable only if the app polls the conflicts API (Automerge), or a conflict surfaced/blocking by default for review (DoltLite)? For a contended status/claim field, surfaced-by-default is the safer fit, which is why DoltLite wins on semantics.
- **Automerge 2** (verified 2026-06-24): pure-Rust crate (trivial embed), production-mature, low-risk. CORRECTION (advisory 2026-06-24): it does NOT lose data; it retains all concurrent values and exposes them via the conflicts API. `get()` returns a deterministic winner by default, so a clobbered claim is detectable but only if the app actively polls (surfacing is opt-in, not automatic). Carries history-bloat managed via compaction. Remains the FALLBACK engine.
- **DoltLite** (verified 2026-06-24, BLOCKED): right on paper (git-style surfaced/blocking merge, relational, git-for-data identity) BUT DoltHub engineer Aaron Son's 2026-04-08 code review documented and reproduced live: (1) concurrent connections silently LOSE COMMITS (a committed hash vanished; "does not correctly implement multiple connections"; no exclusive file access, so silent corruption); (2) durability bug, root ref written before the data it points to is fsync'd (crash = loss); (3) open-path drops errors where "all data in the database has been lost" is silently ignored. Also ~23k lines of agent-written C that no human had read, tests largely self-written; no Rust binding (FFI only); and its multi-machine story is push/pull to doltlite-remotesrv (git-style), NOT live multi-writer, so even working it would not give seamless cross-device sync. Re-evaluate only when the concurrent-write + durability bugs are fixed and verified (that is the gate).
- **Decision rule:** DoltLite is BLOCKED for canonical data until its concurrent-write and durability bugs are demonstrably fixed. Ship multi-place NOW on git-merged JSON/TOML + content-hash IDs (safe, zero new engine, syncs over the existing git remote). If genuine live concurrent-merge is needed before DoltLite is safe, use Automerge (Rust-native, no data loss; wire conflict-surfacing via the conflicts API). Keep StateBackend swappable so DoltLite can drop in later if it earns it. cr-sqlite (SQLite+CRDT) is a footnote third door.
- **Why not one engine for everything:** both store binary git cannot line-merge, so content would stop being readable files in git (cairn's identity). They are the status-merge slot only, never the content store.
