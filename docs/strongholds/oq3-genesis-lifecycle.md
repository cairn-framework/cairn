# OQ3: genesis.md lifecycle on apply

## Methodology

This is an adversarial debate between three lifecycle options for the `research/genesis.md` artefact introduced by C1.a. Each option gets a steelman defender and a steelman attacker; both are weighted equally; the verdict emerges from which option survives the strongest attack rather than which has the strongest case-for. The decisive lens is whether the option respects cairn's two-chain architecture (provenance vs authority), the spec.md change-directory delta semantics, and the actually-implemented behaviour of `cflx.py archive_change` (not just the prescribed behaviour). Where the prescribed semantics and the implemented semantics diverge, the divergence itself is evidence.

## Option A: Codex re-points `nodes` on apply

**Steelman case-for**: Genesis is structurally a piece of research provenance for whatever decision the change embodies. The `nodes` field of a research artefact (per `docs/spec.md` line 411 and the `## Requirements` block at `openspec/specs/artefacts/spec.md` lines 5-15) is supposed to point at *the graph nodes the research informs*. At propose-time the change ID is a placeholder because the actual graph nodes the change touches are not yet decided; the proposal is the act of deciding them. By the time `cflx apply` runs, the design.md has been authored and the spec deltas have been written, so the apply-stage codex agent (per AGENTS.md guardrails) knows exactly which declared items it is touching. Re-pointing `nodes: [<change-id>]` to the actual graph node IDs (e.g. `[saas.api.auth, saas.api.notifications]`) makes the artefact comply with the existing research integrity rule (every research must reference at least one valid node ID, per spec.md line 433) at the moment the artefact transitions from "proposed" to "real provenance about real nodes." This is the same pattern decisions already use: their `nodes` field is authored at propose-time but typically pointed at concrete blueprint IDs by the time the change archives. Option A is the only option that produces a genesis artefact whose `nodes` field will resolve under `cairn scan` once the change merges.

**Steelman case-against**: Re-pointing creates kernel-side hash and reconciliation churn for no payoff. The artefacts spec (lines 22-31) requires the scanner to enforce integrity at scan time, but the `nodes` field of a research artefact is only consulted when the artefact is loaded by the blueprint pointer pattern (`research "./meta/research/api/auth/"`, per spec.md line 241). Genesis-as-elicitation-transcript does not naturally belong under any one node's research pointer, it informs *the change as a whole*, not a single module. Forcing the codex to pick "the" set of touched nodes will produce inconsistent results across phases (some codex runs will pick three nodes, some will pick eight, some will pick the change's primary spec area only) and create a categorical mismatch: research artefacts are supposed to cite nodes that already exist in the blueprint, but a *propose-stage* genesis citing post-archive nodes is citing things that did not exist when the genesis was authored. That is anachronistic provenance: the artefact claims to be informed-by-and-informing things it could not have been, retroactively. Worse: if Option A also implies that the codex agent moves the genesis file from `openspec/changes/<id>/research/genesis.md` to `meta/research/<somewhere>/genesis-<id>.md` to match a node's research pointer, that move both adds a new file to current truth (which the `research` integrity rule would then start enforcing against) and breaks the reproducibility of "what was in the change directory at apply time." The propose-stage transcript is now retroactively tied to nodes the proposer didn't know about.

**Evidence weight**: The cross-check E D3 explicitly invented Option A's seed: *"When the change applies, the apply-stage codex re-points the genesis research's `nodes` at the actual node(s) touched, or moves it to the global provenance index."* So Option A has stronghold-level prior endorsement. **Against**: the existing change directory pattern in spec.md §9.1 (lines 480-498) treats every artefact file in a change directory as a delta with explicit `operation: added|modified|removed|renamed` frontmatter, there is no concept of "the codex updates fields at apply." The delta semantics is *file-level replacement*, not *field-level update*. Re-pointing would invent a new operation type (`field-updated`?) that the spec does not currently have. Phase-2.6's archived directory (the ground-truth example) carries zero research files, so there is no precedent for any per-file mutation during apply. The artefacts spec's "missing required frontmatter fails" scenario (lines 16-20) suggests the loader is strict about field presence and resolution but does not specify a re-pointing path. Net: prior endorsement from one cross-check; no implementation precedent; structural conflict with delta semantics.

## Option B: Move file to global provenance

**Steelman case-for**: Genesis is *not* about a specific graph node. It's about the elicitation conversation that produced a change proposal. The natural home for an artefact-about-a-conversation is the global provenance index (per spec.md line 14: "attaches parsed records to the node or global provenance index"). At archive time, moving `openspec/changes/<id>/research/genesis.md` to `openspec/research/genesis-<id>.md` (or `./meta/research/genesis/genesis-<id>.md`) makes genesis discoverable forever after the change archives. This is exactly how the existing research artefact type works for cross-cutting research: spec.md line 489 shows research at `./meta/research/` (not nested under any one node's directory), and the integrity rule (line 433: "Every research artefact must reference at least one node ID and at least one source ID") supports research that informs multiple nodes. Option B aligns with the spec.md §9.3 archive merge order: research files in change directories with `operation: added` frontmatter are supposed to land in `./meta/research/` at archive (this is the prescribed archive pipeline behaviour, even if the current `cflx.py archive_change` doesn't yet implement artefact-delta processing). And it's the only option that produces a consistent answer to "where do I find the genesis for change X six months after it archived?", under the global provenance index, by ID, the way decisions, sources, and other research already work.

**Steelman case-against**: Option B inherits all of the bookkeeping cost of artefact-delta processing without any of the existing infrastructure to do it. The `cflx.py archive_change` implementation (lines 462-501) currently does `shutil.move(str(change_dir), str(archive_dest))` and `_update_specs_from_change()`, it processes spec deltas but not artefact deltas. Adopting Option B forces cflx.py to learn how to process at least *one* artefact-delta type (genesis-as-research) before any of the others. That's a phase-scope creep: C1.a is supposed to be a skill-body edit (cross-check E R4), not a cflx.py archiver upgrade. Worse: where does global genesis go? The repo today has no `./meta/research/` directory (it has `openspec/specs/`, `openspec/changes/`, `openspec/registries/`). Spec.md line 489's `./meta/research/` is *prescribed* layout, not implemented layout. C1.a would be the first artefact to need a `meta/` subtree that doesn't exist, forcing C1.a to either invent the subtree (ballooning scope) or pick a non-spec-aligned home like `openspec/research/` (which then becomes a precedent for other artefact types when phases 8-10 implement the prescribed `meta/` layout, leading to a future migration). And if genesis lives at `openspec/research/genesis-<id>.md`, then querying genesis by change-id requires either parsing all genesis files or building a `genesis-<id>` → `<change-id>` index. The change-directory-relative path (`openspec/changes/archive/<id>/research/genesis.md`) is *already* a globally unique, ID-keyed path, Option C gets that for free.

**Evidence weight**: The case-for cites three things directly: spec.md §3.1 provenance chain semantics (genesis is provenance), spec.md §8.5 research schema (supports multi-node research), and the prescribed archive merge pipeline (§9.3) which would in principle move research deltas to `./meta/research/`. **Against**: zero current implementation of artefact-delta processing in `cflx.py archive_change`. Zero existing `./meta/research/` directory in the live repo (verified via `ls`-style inspection of repo layout: `openspec/`, `docs/`, `src/`, `test/`, no `meta/`). Phase-2.6's archive carries no research; the ground-truth pattern is "change directories carry no research today, archived directories carry no research today." Option B is the spec-aligned answer to a question that the spec has never had to actually answer in production. Net: defensible by spec semantics; offensive by implementation reality.

## Option C: Leave in change directory permanently

**Steelman case-for**: Option C is what the actual archive implementation does today. `cflx.py archive_change` line 491 is `shutil.move(str(change_dir), str(archive_dest))`, the entire change directory, including any subdirectories (proposal.md, design.md, tasks.md, specs/, and any new research/), moves wholesale to `openspec/changes/archive/<change-id>/`. No special handling. No per-file lifecycle. The genesis file written by the cflx-proposal skill at `openspec/changes/<id>/research/genesis.md` becomes `openspec/changes/archive/<id>/research/genesis.md` after archive. That path is globally unique (the change-id is unique; archived change directories are date-prefixed in spec but currently un-prefixed in cflx.py impl, both forms preserve uniqueness). Discovery is easy: "why was decision X made" → look up the change that introduced X → cd into `openspec/changes/archive/<that-change>/research/genesis.md`. The change directory is an *eternal record-of-process*, not a tombstone, it preserves not just what was decided but how the decision was elicited. This matches the `## Summary` + `## Transcript` body shape (cross-check E R7): the transcript is *of the elicitation that produced the change*, so its natural home is *with the change*, not detached from it. Crucially, Option C requires zero changes to AGENTS.md, zero changes to cflx.py, zero new conventions for `meta/research/`, and zero kernel-side reconciliation churn. The skill writes the file; the existing archive bulk-move handles everything else. C1.a stays scoped to "a skill-body edit plus a one-line conventions.md note" exactly as the cross-check E R4 prescribed.

**Steelman case-against**: Option C makes genesis structurally invisible to the cairn graph. The scanner (per `openspec/specs/artefacts/spec.md` Requirement 1) loads research artefacts via blueprint pointers (`research "./meta/research/..."` per spec.md line 241). A research artefact at `openspec/changes/archive/<id>/research/genesis.md` is not under any blueprint's research pointer, so the scanner never sees it; it never appears in `cairn rationale`, `cairn research`, or `cairn neighbourhood`. That breaks the entire purpose of recording elicitation provenance: the canonical "why was decision X made" lookup is `cairn rationale <node>` (spec.md line 706), and Option C produces a class of research that cairn cannot rationale. Worse: Option C creates two distinct categories of research with the same artefact type, "real" research at `./meta/research/` that the scanner sees, and "ghost" research in archived change directories that only humans see. The two-chain architecture (provenance: source → research → decision) requires the *whole chain* to be queryable via the framework. Option C amputates the chain at the genesis link: the decision is in the graph, but the research that informed it is not. The cross-check E observation that `nodes: [<change-id>]` uses the change ID as a placeholder reveals the real problem, a change ID is not a graph node. Option C says "leave it that way forever," which means genesis artefacts permanently violate the research integrity rule (line 433: "every research artefact must reference at least one node ID"). They would surface as findings if the scanner ever tried to load them, which by Option C's design it never will. The artefact spec is silent on the location, but it is *not* silent on the integrity rule, and Option C produces artefacts that pass-by-virtue-of-not-being-loaded.

**Evidence weight**: This is where the evidence asymmetry is sharpest. **For Option C**: phase-2.6's archived directory contains zero research files, the entire body of past phases archived without any research subdirectory. The cflx.py archive implementation is bulk-move-only, no artefact delta processing exists, and adding it is out of C1.a's scope. CLAUDE.md "Where things live" table treats `openspec/changes/archive/<phase>/` as the historical record of all phase content, not a tombstone. The cross-check E open question itself recommends "leave it; archive carries it through" as the simplest answer. **Against Option C**: the artefacts spec integrity rule technically applies to research artefacts wherever they live, and Option C produces research the scanner never loads. But the counter-counter: *the scanner doesn't load anything in `openspec/changes/archive/`* in the current implementation, archived change directories are out-of-band by design. Phase-2.6's archived `specs/` directory is consulted only by `cflx.py validate`, never by the cairn Rust scanner. Genesis-in-archived-change-dir is in exactly the same epistemic class as `proposal.md`, `tasks.md`, and `design.md`: forever-reachable by humans and codex agents that grep `openspec/changes/archive/`, never reached by `cairn scan`. The amputation argument is real but it's an amputation that already exists for everything in the change directory.

## Cross-cutting considerations

### Two-chain architecture compatibility

The provenance chain is `source → research → decision`. Genesis is research. The chain is supposed to be discoverable via the cairn graph: `cairn rationale <node>` returns "accepted decisions attached to this node and its direct neighbours, plus the research and sources informing them" (spec.md line 706). Genesis must therefore be reachable from a decision via `informed_by` to participate in the chain. **Option A** preserves chain discoverability *if* the codex re-points `nodes` and *if* the corresponding decision artefact lists genesis under `informed_by`. **Option B** preserves chain discoverability if the genesis lands at a scanner-visible path (`./meta/research/`) and the decision references it. **Option C** preserves chain discoverability *only* if humans look up the change directory by hand, or if cairn is taught to also load archived change-directory research, or if every change's genesis is also referenced by `informed_by` in a decision artefact that does live in `./meta/decisions/`. The spec's two-chain architecture is most cleanly served by Option B; Option A is half-served (genesis is discoverable but its `nodes` field has been retroactively rewritten); Option C is unserved by the framework but human-discoverable via a parallel path.

### Archive semantics

What does `openspec/changes/archive/<phase>/` mean? Three interpretations the brief listed:
1. *Tombstone-of-decision*: the directory is a marker that the change happened, with proposal/tasks/design as historical justification. Option C-friendly.
2. *Eternal record-of-process*: the directory is the full record of the elicitation, design, and execution of the change, preserved for institutional memory. Strongly Option C-friendly.
3. *Strictly historical for the change, not for any artefact*: the directory is closed once archived; nothing in it is consulted by current-truth tooling. Option C-unfriendly because genesis becomes inaccessible to the framework.

Phase-2.6's archive directory is interpretation (2): it carries `proposal.md`, `tasks.md`, `design.md`, `ALLOWLIST.md`, and `specs/`. Specifically, ALLOWLIST.md (a phase-specific working file) survived archive, strong precedent that change directories are eternal record-of-process, not tombstone. The cflx.py archiver also implements interpretation (2): bulk-move preserves everything. The artefacts spec is silent, but the change-system spec (`openspec/specs/changes/spec.md` lines 28-34) describes archive as a move-with-spec-update, not a content-prune. **Net: archive semantics in this repo is unambiguously eternal-record-of-process. Option C aligns with the existing semantics; Options A and B propose new lifecycle behaviour that the existing implementation does not support.**

### Hash and reconciliation impact

The reconciler (per CLAUDE.md "Architecture: two chains") reconciles the blueprint-declared map against scanned reality. Artefact-side reconciliation includes the integrity rules: orphan research surfaces as info-level (line 433); decisions with unresolved `informed_by` produce rationale tensions (artefacts spec scenario 26-31). 

- **Option A** triggers reconciliation churn: re-pointing `nodes` at apply-time means the genesis artefact's `nodes` set changes between propose-stage and post-archive-stage. If the scanner loaded the genesis at any propose-time scan (e.g., during `cflx accept` verification), the post-rewrite version is a different artefact-by-content even though it has the same ID. That breaks the interface-hash invariant for research (spec.md mentions interface hashes for code/contracts but the principle generalises). It also creates a transient window where genesis's `nodes` field is `[<change-id>]` (which is not a valid blueprint node ID, since change IDs are not blueprint IDs), causing the integrity check to fail until the codex finishes re-pointing.
- **Option B** triggers a one-time reconciliation event at archive: the genesis file appears at `./meta/research/` from the scanner's perspective, just like any new research artefact would. This is the *intended* archive flow per spec.md §9.3 (ADDED operations append new content). Reconciliation accepts the new artefact cleanly. No hash churn.
- **Option C** has zero reconciliation impact because the genesis is never loaded by the scanner. The reconciler doesn't see it; it can't churn over it. This is either a feature (zero cost) or a bug (the chain is incomplete) depending on which side of the two-chain compatibility argument wins.

### Discoverability

"Why was decision X made" six months after the change archives:
- **Option A**: `cairn rationale <node>` returns the genesis (because `nodes` was re-pointed at the actual node) plus the decision's `informed_by` link points at it. **Best framework-side discoverability**, assuming the re-point happened correctly.
- **Option B**: same as A in framework-side discoverability *if* genesis lands under a scanner-visible research pointer; if genesis lands at a global-provenance path (`./meta/research/genesis/`) but no blueprint pointer exists for that subdirectory, the scanner won't see it either. Requires a `research "./meta/research/genesis/"` pointer somewhere, most naturally on a synthetic root or system-level node.
- **Option C**: framework-side discoverability is zero. Human-side discoverability is high: `find openspec/changes/archive -name genesis.md | xargs grep <decision-name>` works; navigating from a decision artefact's `informed_by: [genesis-<change-id>]` to `openspec/changes/archive/<change-id>/research/genesis.md` works by ID convention. Codex agents during apply can read it the same way. 

The asymmetry: Option A and B win framework-side discoverability; Option C wins zero-cost simplicity but loses framework reach.

## Verdict

- **Choice**: **Option C with a hybrid escape hatch.**
- **Reasoning**: Option C survived the strongest attack (the two-chain amputation argument) because the attack assumes the scanner reaches into archived change directories, which it has never done and never will under current cairn semantics. The amputation argument is real but it is also true of every other artefact in archived change directories, `proposal.md`, `tasks.md`, `design.md`, the `ALLOWLIST.md` from phase-2.6 are all framework-invisible. Genesis joins them. The provenance chain is preserved in the *human and codex-readable* sense, just not the *scanner-loadable* sense. The decisive evidence is that **Option B requires implementation work the cflx.py archiver does not yet do** (artefact-delta processing) and creates a precedent where every new artefact type proposed in a change directory needs a per-type lifecycle decision before it can ship. Option A requires the codex to invent field-level update operations that the change-system spec does not have. Option C requires zero new infrastructure and matches the existing bulk-move archive behaviour exactly. The "hybrid escape hatch" caveat: when (not if) phase 8+ implements the prescribed `meta/research/` subtree and the cflx.py archiver gains artefact-delta processing, genesis can be migrated to Option B retroactively without breaking anything (the change directory's genesis becomes the source-of-truth for the migration; the migration produces `./meta/research/genesis/` entries from each archived change). This is forward-compatible with Option B without paying its cost today.
- **Confidence**: **medium-high.** High on the evidence: phase-2.6's archive carries zero research, cflx.py archive_change is bulk-move-only, the change-system spec treats archive as eternal-record-of-process. Medium because the two-chain architecture argument is genuine; if cairn ever grows a "scan archived changes for provenance" mode, Option C's choice will need revisiting.
- **What would flip this**: 
  - If `cflx.py archive_change` grows artefact-delta processing as part of any phase before C1.a ships (most likely candidate: phase 8 summariser, which generates artefacts), then Option B becomes free and should be adopted, the same delta-processing code path that handles summariser-generated decisions/research will handle genesis. 
  - If a future phase adds a `cairn rationale` mode that includes archived-change provenance (e.g., `cairn rationale <node> --include-archived`), Option C remains correct because the scanner reach extends to archived dirs. 
  - If the scanner ever starts treating change-directory artefacts as first-class scannable content (a major shift in current-truth semantics), Options A and B become the only valid answers and Option C is wrong.

## Implementation requirements

For Option C to be the durable answer, three concrete artefacts need to land:

### 1. SKILL.md (`.claude/skills/cflx-proposal/SKILL.md`)

```yaml

id: genesis-<change-id>

date: <YYYY-MM-DD>      # propose-stage date
sources: []
informed_by: []
type: genesis
---

## Summary
[3-6 bullets, mirroring existing Premise / Context output]

## Transcript
[Full elicitation Q/A turns]
```

The skill body must explicitly note that `nodes: [<change-id>]` is a placeholder using the change ID rather than a blueprint node ID, and that the genesis artefact lives at the change-directory-relative path forever (no migration on apply, no migration on archive).

### 2. AGENTS.md (`/Users/george/repos/cairn/AGENTS.md`)

**No edit required.** This is the load-bearing simplicity of Option C: the apply-stage codex agent does not author, modify, move, or rewrite genesis. The codex agent may *read* genesis to understand elicitation context (the same way it reads `proposal.md` and `design.md`), but does not touch it. Confirming cross-check E R4's "AGENTS.md update: NONE."

### 3. cflx.py (`.claude/skills/cflx-proposal/scripts/cflx.py`)

**No edit required.** The existing `shutil.move(str(change_dir), str(archive_dest))` carries `research/genesis.md` with the rest of the change directory at archive. No validator change (per cross-check E D7: genesis is convention-encouraged, not enforcement-required).

### 4. conventions.md (`/Users/george/repos/cairn/openspec/conventions.md`)

One-line addition documenting the convention:

> **Genesis transcript.** Proposal authoring via the `cflx-proposal` skill writes the elicitation transcript to `openspec/changes/<id>/research/genesis.md` with id `genesis-<change-id>`. The file lives in the change directory, archives with the change directory, and is not loaded by the cairn scanner. It is human-and-codex-readable provenance for the change's elicitation history.

This makes the convention discoverable to future authors, codex agents reading conventions during apply, and any future phase that wants to migrate genesis to a scanner-visible location.

### 5. (Forward-compat note) Future-phase docket

Add a one-line note in `docs/strongholds/oq3-genesis-lifecycle.md` (this file) and/or in the eventual phase-9 brownfield rescope: when artefact-delta processing lands in cflx.py (likely phase 8 summariser or later), revisit genesis lifecycle. Migrate to Option B if and only if (a) the artefact-delta pipeline is in place, (b) `./meta/research/` exists in the live repo, and (c) the scanner is configured to load `./meta/research/genesis/`. Until all three hold, Option C is correct.
