# Node model

## What this is

How getcairn.dev structures the things in a model. A node is the universal primitive: the system root, every subsystem, every leaf component is a node. Each node carries a description, a type, children, budgets, properties, attachments, and interfaces. Properties are tagged when generated by AI. The action affordances on a node are uniform: `Decompose with AI`, `Generate Requirements`, plus delete.

For how nodes are scored, see [05-completeness-and-causality.md](./05-completeness-and-causality.md). For how to interact with a node from the palette, see [06-command-palette.md](./06-command-palette.md). For their term-by-term mapping to our concepts, see [07-ontology-comparison.md](./07-ontology-comparison.md).

## ID scheme

The captured fixture uses these IDs (**from screenshots 06 to 16**):

- `SYS-OFSUSR` for the system root.
- `SUB-USV`, `SUB-PWR`, `SUB-LARS` for subsystems (USV Platform, Power Generation & Distribution, Launch & Recovery System).
- `REQ-001`, `REQ-003`, `REQ-006` for requirements (**from screenshots 21, 23, 24**).

The pattern is `{TYPE}-{ABBREV}` for nodes and `REQ-{NNN}` for requirements (**inferred** from the small sample). ID stability across rounds is reported by the prior scout (**verified**), but not directly observable from a single-session capture.

## Node types

From the surface and the prior research:

- **System** (the root). Holds the overall brief, top-level interfaces, and links to all subsystems.
- **Subsystem.** A child of the system or another subsystem. Carries its own brief, properties, budgets, interfaces, children.
- **Requirement.** A "shall" statement attached to a node, with verification metadata and acceptance criteria. (**verified** via docs reference.)
- **Interface.** A typed connection between two nodes, carrying a payload signature (`HVDC power + fibre-optic data`, `Ethernet / IP`) and a signal count. (**from screenshot 07**.)
- **State machine.** Mentioned in the status-bar count (`0 states`, **from screenshot 11**) and as a generation target in the command palette (`Create a state machine for ...`, **from screenshots 05 and 24**). Not directly captured beyond the count.
- **Visual / Attachment.** Files attached to a node. Visuals appear to be a renderable subtype; attachments are general files. (**from screenshot 07**.)

The Type registry tool (left-rail TYPES button) is **inferred** to be the surface where these node types are defined or extended.

## Node detail panel anatomy

When a node is selected, the right-side detail panel shows (**from screenshots 09 and 10**):

### Header
- Type chip and node ID (e.g. `SUB-USV · Node`).
- Title (e.g. `USV Platform`).
- Close X.

### Description
Plain prose paragraph, truncated with `Show more` when long.

### Type and children
- `TYPE: Subsystem`.
- `CHILDREN: 0 child nodes`.

### Budgets

A first-class concept distinct from properties. The captured panel shows two budgets (**from screenshot 09**):

```
BUDGETS
  Mass    0 / 8,000 kg    (no budget)
  Power   0 / 15 W        (no budget)
```

The "0 /" prefix shows current consumed, the post-slash shows the cap. The "(no budget)" tag indicates the budget is declared but not yet spent against.

Budgets aggregate up the tree: at the system root, `SYSTEM BUDGETS` shows `Mass: 800 kg / Power: 18 W` (**from screenshot 06**). The mechanism is **inferred** to be a sum-up-the-tree rollup, with each subsystem contributing its allocated share to the parent's total. How rollups are computed when a child has unset budgets is **unknown**.

Budgets are a distinct concept from properties: they are a structural envelope for the node, not a value of the node. This distinction is worth preserving if we adopt anything from this layer.

### Properties

Each property has an icon, name, value, unit, category, AI-tag (when present), and short description (**from screenshot 09**):

```
◎ Mass Budget       8000 kg     Physical · ai · Estimated displacement
⚡ Power Budget      15 W        Electrical · ai · Estimated vessel hote...
⚙ Max Transit Speed 10 m/s      Performance · ai · Estimated max transi...
   Max Transit Sea State  6      · ai · Maximum sea state for transit op...
```

Notable structural features:

#### Properties and AI tagging

Every property in the captured node carries the `· ai ·` token in its category line. This signals the value was AI-generated, not authored by the user. Authored values would presumably lack the tag (**inferred**, not directly verified; none of the captured properties are user-authored, so the absence-state is not confirmed).

This is the **highest-leverage observation in the node model**. By tagging every value as authored or generated, the user knows at a glance which numbers are their commitments and which are AI proposals awaiting validation. Maps onto our reconciler's job of distinguishing the authored blueprint from the generated map. See [08-borrow-list.md](./08-borrow-list.md#5-explicit-ai-vs-authored-tagging-on-property-values) for the borrow rationale.

#### Properties as a typed list

Properties have categories (`Physical`, `Electrical`, `Performance`) and units (`kg`, `W`, `m/s`). The category set is **inferred** to be a fixed taxonomy (closed) but the underlying registry is the Types tool.

#### Suggested-property chips

Below the typed property rows, a row of "+" chips shows AI-suggested additional properties (**from screenshots 09 and 10**):

```
+ Nominal Power   + Nominal Voltage   + Max Range
+ Data Rate       + Protocol          + Peak Power
                  Show all (13)
```

The `Show all (13)` chip implies a full library of 13 candidate properties is available for one-click addition. This is a UX nudge: the AI has staged candidates and the user accepts or ignores. Adopt-or-discard at the property level rather than batch.

The mechanism for generating the candidate list is **unknown** but plausibly a per-type schema plus context-aware ranking.

### Attachments

```
ATTACHMENTS                                                    +
   No files attached
   + Attach file
```

(**from screenshot 10**.) Attachments are first-class but optional.

### Interfaces

```
INTERFACES (0)
   No interfaces yet
   + New Interface
```

Interfaces on a node are inbound and outbound connections to siblings. At the project root the interfaces section shows two captured connections (**from screenshot 07**):

```
Power Generation & Distribution → ROV Vehicle              HVDC power + fibre-optic data    4 signals
Autonomy, Control & Mission Payload → Communications & Data Link    Ethernet / IP    4 signals
```

Each interface has a directed edge, a payload type chip, and a signal count. Whether signals are sub-elements of an interface (lower-level wires) is **inferred** but plausible.

### Action buttons

Two large action buttons sit at the bottom of every node detail panel (**from screenshot 10**):

- **`◆ Decompose with AI`** (filled black). Generates child nodes for the current node.
- **`◆ Generate Requirements`** (outlined). Generates requirements attached to this node.

Plus a smaller delete button (red outline). The diamond glyph is consistent across AI-driven actions; see [06-command-palette.md](./06-command-palette.md) for the same glyph in the palette.

The action set is **uniform across node types** as far as the captured screenshots show. Any node can be decomposed and any node can have requirements generated. The actions are not contextual to type but to position: a leaf node generates child decomposition; a parent generates requirements.

## What this implies for our model

(Mapping is in [07-ontology-comparison.md](./07-ontology-comparison.md); architectural implications here.)

- Their node is **single-graph**: every primitive is a node with the same shape. We have a typed-schema kernel where contract, decision, todo, research, review, and source are direct artefact types with different schemas. Theirs is more uniform but less expressive.
- Their **AI tagging on property values** is closer in spirit to our scanner's job of distinguishing authored from generated state. We currently do not surface that distinction at the property level inside the map.
- Their **budgets-as-distinct-from-properties** maps onto our spec's separation between contract obligations and observed values. Our blueprint can express budget-style constraints; we do not currently surface them as a separate first-class category in the UI.
- Their **fixed first-pass shape** (1 root + 4-7 subsystems + key interfaces + brief) is a generation default, not a schema rule. We should not adopt the constraint, but the default-shape pattern is borrowable for a future webui entrypoint.
