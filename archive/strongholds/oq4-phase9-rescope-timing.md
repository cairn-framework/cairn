# OQ4: Phase 9 rescope timing

## Methodology

I treat each timing option (A: rescope-before, B: rescope-after, C: concurrent) as a candidate against the integrated plan's Wave 4 placement and stress-test it against four discrete adversarial axes: (1) what cflx workflow primitives the option commits to (`apply`, `accept`, `validate`, `archive`, proposal-only edits); (2) how the option degrades if assumptions move during execution (Bundle A surfaces a scope shift; the rescope itself uncovers Phase 9 gaps; phase-9.0-tests applies before the rescope); (3) what evidence the live tree provides (does Phase 9's proposal already contain any of the absorbed scope, does Phase 2.6's archived apply set a precedent for cross-phase proposal edits); (4) what fails silently if the option is wrong. Each option gets a steelmanned defender's case and a steelmanned attacker's case. The verdict synthesises across the three.

I read the cross-check-A bundle finding, the cross-check-integrated Pattern 3 decision, the live Phase 9 proposal/design/tasks/specs, the phase-9.0-tests proposal, the archived phase-2.6 proposal (for precedent), the cflx-proposal SKILL.md (for what "rescope" means as a verb), and the on-disk `cflx.py validate --help` to determine whether validate operates per-change or per-corpus. The decisive piece of evidence is `cflx.py validate phase-9-brownfield --strict` (currently passing) operating on a single named change-id, meaning Phase 9's transient validate state during a rescope window does not gate any other phase's accept.

## Option A: Rescope BEFORE Bundle A ships

**Steelman case-for**: Author the rescope-update first, then build Bundle A against a Phase 9 proposal that already declares its dependence on Bundle A's primitives. This buys two correctness goods. (1) Bundle A's design.md author has a written downstream contract: Phase 9's rescoped proposal cites the suggested-edges file format, the trace sidecar location, and the validate-strict gate by name, Bundle A's design.md can quote those names verbatim instead of inventing them and hoping Phase 9 will use the same vocabulary. This removes a cross-phase naming race. (2) The cflx-proposal lifecycle in this repo treats proposal authoring and apply as separable steps (per SKILL.md line 455's "Updates proposal" verb), so authoring a rescope before Bundle A's apply does not block Bundle A's apply on the rescope's correctness. The rescope is docs-only; if it has a typo, it gets fixed in a follow-up. Phase 2.6's archived apply commit (6a54169) is direct precedent: that commit edited four downstream phases' files inside the phase-2.6 apply itself, so cross-phase proposal-state changes are normal practice. Authoring the rescope first compresses the cross-phase narrative into a single declarative state, which improves reviewability of Bundle A's PR (reviewers can read Phase 9's rescoped proposal alongside Bundle A's design.md).

**Steelman case-against**: Bundle A is itself unstable. Cross-check A R2 says the suggested-edges file is `suggested-edges.json` or `suggested-edges.toml`; D4 leaves the format choice for Bundle A's design.md authoring pass; open question 5 explicitly punts JSON-vs-TOML to that pass; D7 is uncertain whether the new capability area is `provenance-foundation` or `cflx-trace`. If the rescope cites the file as `suggested-edges.json` and Bundle A then chooses TOML, the rescope is wrong on the day Bundle A merges. The rescope would need a re-update commit, doubling the proposal-edit work. This is the *re-rescope risk*: committing the downstream contract before the upstream surface is fixed. Worse, if Bundle A's accept gate (cross-check A R3) ends up named `CFLX-VALIDATE-001` instead of `CFLX-ACCEPT-001` (open during D5), the rescope's reference to the gate is also wrong. In short: rescoping-before treats Bundle A's design as already fixed, when in fact Bundle A's cross-check explicitly defers four naming decisions to its own design.md authoring pass. The load-bearing constraint Option A breaks: **a downstream-contract document that names not-yet-authored upstream identifiers is structurally fragile.** Each defer-resolved identifier is a future patch to the rescope.

**Evidence weight**: Cross-check A explicitly defers (file-name format, capability-area name, gate-error code) to Bundle A's own design.md. This evidence directly refutes Option A. Phase 2.6's precedent (editing other phases' files inside the apply) is *for* Option A on the precedent axis but does not address the upstream-instability axis (in 2.6 the upstream, terminology rename itself, was fully designed before the cross-phase edits, because it was the apply commit; here Bundle A's design has not started). Phase 9's current proposal validates `cflx.py validate phase-9-brownfield --strict` as `Validation passed`; rescoping it would temporarily break that pass (validate-fail is licensed by CLAUDE.md but is real). Verdict: Option A is the most aggressive on coupling and the most exposed to upstream churn. Defensible if Bundle A's design.md authoring is so tightly outlined that the deferred decisions are token-level not architectural; in practice cross-check A's open questions are token-level (JSON vs TOML, gate name) so this is partially recoverable, but a re-rescope commit is the realistic outcome.

## Option B: Rescope AFTER Bundle A ships

**Steelman case-for**: Bundle A's apply will reveal the actual file format, the actual gate-error code, the actual capability-area name, and the actual sidecar location. Once those are concrete, the rescope cites them by their final identifiers, no naming race, no re-update cycle, no fragile downstream-contract patches. This option also matches the cflx workflow's natural information flow: code lands first, downstream documents reflect what code actually does. Phase 9's current proposal is internally consistent and currently passes `cflx.py validate phase-9-brownfield --strict`, so rescoping later means Phase 9's validate-pass status persists across Bundle A's apply window, Bundle A's accept and archive run while Phase 9 continues to pass strict validate, removing one entire class of "did anything else break?" question from Bundle A's verification battery. The rescope-after window is also when the orchestrator has the most information: Bundle A's actual implementation may surface design refinements (e.g., the queueing format may grow a `provenance` field that Phase 9's suggest engine needs to populate), and rescoping after captures those refinements without an authoring loop. The integrated plan's Wave 4 placement (Pattern 3 paragraph 3: "after Bundle A is in") is exactly this option, and was reached by an integrator synthesising six independent cross-checks; that's stronger consensus than any single bundle's preference.

**Steelman case-against**: The rescope-window cost is real even if validate-pass holds. Wave 2 ships Bundle A and Bundle B in parallel; Wave 3 ships Bundle C, Bundle E, and the spec-validate cleanup sweep in parallel. *During those waves*, Phase 9's proposal claims a scope (deterministic-extraction-only) that the orchestrator already knows is wrong. Anyone reading Phase 9's proposal mid-Wave-2 sees a Phase 9 that has no suggest engine, no interview runner, no templated authoring, a Phase 9 that does not exist. The integrated plan's documented intent is invisible from inside Phase 9's own files. This is a documentation-staleness window of unbounded duration (Bundle A's apply timeline is not fixed); it lasts however long Bundle A takes plus however long the rescope authoring takes. If a new contributor reads the project during that window, they get a wrong picture of the next phase's scope. More technically: phase-9.0-tests' proposal references "ten acceptance-criterion scenarios across four requirements" derived from Phase 9's *current* proposal. If Phase 9 rescopes after Bundle A and adds new requirements (suggest-engine acceptance, interview-runner acceptance, templated-authoring acceptance), phase-9.0-tests is already-authored against the wrong scope; phase-9.0-tests itself needs updating to match the rescoped Phase 9. The integrated plan acknowledges this in its sequencing rule "Phase 9 rescope → phase-9.0-tests apply", but if phase-9.0-tests is already in its proposal-authored state with an `#[ignore]` count tied to the un-rescoped Phase 9, that authored proposal is also stale during the window. Net: Option B trades upstream-instability risk (Option A's vice) for documentation-staleness drift (Option B's vice). The load-bearing constraint Option B breaks: **the project's declarative state should not lie about the next phase's planned scope.**

**Evidence weight**: phase-9.0-tests' proposal already references Phase 9's current ten-scenario count. Cross-check A's "in no scenario should C8.c ship before C8.b" supports Bundle A first as a hard constraint, but does not say *the rescope has to wait for Bundle A to ship*. The two are separable: Bundle A's apply must precede C8.c apply, but the rescope (a docs-only proposal-update commit) need only precede phase-9.0-tests apply, not Bundle A's apply. Phase 2.6's precedent is mute on this question because phase 2.6 was its own apply, not a downstream rescope. The integrated-plan Wave 4 placement is the single strongest piece of evidence for Option B (Pattern 3, integrator decision 3, sequencing rule 4 in cross-check-integrated). But that placement is also a default, not a derivation; the integrator says it's a docs-only commit "slotted between Bundle A's apply and phase-9.0-tests' apply" without explicitly considering whether earlier slotting is feasible. So the Wave 4 placement is a *suggested* timing, not a *forced* one.

## Option C: Rescope CONCURRENT with Bundle A

**Steelman case-for**: Concurrent shipping minimises the documentation-staleness window. Once Bundle A's first commit lands, the rescope-PR is open in parallel; the rescope cites Bundle A's evolving identifiers as Bundle A's design.md fixes them. Both reach merge near-simultaneously, so the wrong-Phase-9 window is shortest. This option treats the rescope as a coordination artefact (per the strongholds-pattern in this repo: stronghold files are exactly this kind of coordination doc) rather than as a sequenced commit. The integrated plan's "Bundles A and B ship in parallel" decision (integrator decision 4) demonstrates the orchestrator is comfortable with parallel work-streams when they don't share files; the rescope-PR doesn't share files with Bundle A (it edits `openspec/changes/phase-9-brownfield/*` only). So adding a third parallel stream during Wave 2 is structurally consistent with the existing parallelism model. CLAUDE.md's "intermediate broken states on a feature branch are acceptable" further licences the rescope-PR sitting in a partially-coherent state during Bundle A's iteration; it doesn't have to merge until Bundle A's identifiers are fixed.

**Steelman case-against**: Bundle A's design.md is itself authored under cross-check A's deferred decisions (D4, D5, D7, Q5, Q6); those decisions might shift during Bundle A's design.md authoring pass. Concurrent rescope means the rescope's draft tracks a moving target. Worse, the orchestrator now has two open PRs in the same conceptual area (Bundle A and Phase 9 rescope) with mutual dependencies; coordination overhead is real. If Bundle A's first commit chooses TOML for the suggested-edges file and the rescope-draft says JSON, the rescope-PR has to update; if Bundle A then chooses JSON in commit 4 (a reversal), the rescope reverses too. This is *scope-discovery churn*, the worst-of-both-worlds case from the original framing. Bundle A is not a one-commit phase per cross-check A's net-LOC estimate (~550-1000 LOC across four sub-components plus a new capability area); that's a multi-day-to-multi-week implementation timeline depending on agent throughput. A rescope-PR sitting open for that timeline accrues merge conflicts with main as other Wave 2 / Wave 3 phases land. Operating in a worktree (per cflx workflow) does not eliminate the merge-back cost; it defers it. The load-bearing constraint Option C breaks: **two PRs whose correctness depends on each other's evolving identifiers create a coordination problem that has no clean serialisation point.** The orchestrator becomes the live conflict resolver between two streams that should be sequential.

**Evidence weight**: There is no precedent in the archive for concurrent docs-and-code PRs with mutual-identifier dependencies. The closest precedents (phase-2.6 cross-phase edits inside the apply commit; phase-7.5b cleansing-splits) are both atomic-within-one-apply, not separate-but-concurrent-PRs. The cflx workflow primitive is a worktree-per-phase; running two phases' worktrees concurrently is structurally supported (the integrator confirms Bundle A and Bundle B run in parallel) but the rescope is not a phase, it's a proposal-update commit; mixing those primitives is novel ground for this repo. The integrated plan does not endorse Option C; it explicitly slots the rescope after Bundle A.

## Cross-cutting considerations

### cflx workflow constraints

"Rescope" in this repo's cflx vocabulary is a proposal-update commit, i.e., a normal git commit editing files in `openspec/changes/phase-9-brownfield/{proposal,design,tasks}.md` and `openspec/changes/phase-9-brownfield/specs/brownfield/spec.md`. It is not `cflx apply`, `cflx accept`, or `cflx archive`. The cflx-proposal SKILL.md confirms (line 455): the verb is "Updates proposal", not "Re-applies" or "Re-archives". The integrated-plan Wave 4 sequencing rule explicitly says "docs-only change (proposal/design/tasks edits + spec delta updates); it doesn't change Phase 9's implementation start." So in cflx terms, the rescope is just a commit on dev (or on a short-lived branch). It does not invoke any cflx pipeline stage. This is decisive for the timing question: a normal commit can land at almost any wallclock moment without lifecycle-state contention.

### Validation timing

`cflx.py validate` takes a single `change_id` argument (`omit for all`). Running `cflx.py validate phase-9-brownfield --strict` currently passes. During the rescope window, validate may temporarily fail (new requirements declared without scenarios; new spec-delta sections added without `## Purpose`; etc.). Three observations: (1) Bundle A's accept gate runs `cflx.py validate phase-7.6-ai-provenance-foundation --strict`, not against phase-9-brownfield, so Phase 9's validate state is irrelevant to Bundle A's accept; (2) the verification battery in CLAUDE.md is invoked per-phase, not corpus-wide, so a rescope-in-progress on Phase 9 does not gate any other phase's accept; (3) `cflx.py validate` (no change_id) does corpus-wide validation but is not part of the standard verification battery, only `cflx.py validate <phase> --strict` is. So the cost of Phase 9 failing validate during the rescope window is *only* the cost of running `cflx.py validate phase-9-brownfield --strict` and seeing red. Nothing else breaks. CLAUDE.md "intermediate broken states on a feature branch are acceptable" further licences this.

### Cflx accept gate

If Bundle A's first commit hits cflx accept while Phase 9's proposal is in a rescoping state, nothing breaks. The accept gate's verification battery operates on the change-id under accept (Bundle A's), not on the entire openspec/changes/ tree. The accept gate does not care about other phases' validity. This removes one of the original framing's stated risks for Option B.

### Process precedent from archived phases

Phase 2.6 (archived merge `3f15946`, apply commit `6a54169`) is the strongest live precedent. That apply commit edited proposals/tasks/designs of phase-3, phase-4, phase-5, phase-10, all downstream phases, inside the same commit as phase-2.6's own implementation work. The precedent says: cross-phase proposal edits inside one apply are normal. It does not, however, address the question of whether the rescope-update should be its own commit or batched with Bundle A's apply. Phase 2.6 batched cross-phase edits with the apply because the apply itself was the rename; here Bundle A is not the rescope, so the precedent suggests batching is feasible but does not forbid separation. There is no precedent in the archive for a *standalone* docs-only rescope commit between two phases.

### Sweep phase interaction

The spec-validate-cleanup sweep ships in Wave 3 (parallel with Bundle C and Bundle E). It targets the residual failing specs that no bundle absorbs. Phase 9's spec at `openspec/changes/phase-9-brownfield/specs/brownfield/spec.md` is currently passing strict validate (per the live invocation result), so it's not in the sweep's scope. The sweep does not touch Phase 9. The rescope and the sweep are independent. (One small interaction worth flagging: the sweep's frontmatter-fix pattern, if it lands a `## Purpose` precedent for change-resident specs, might inform the shape of the rescope's added requirements; but that's a stylistic alignment issue, not a sequencing constraint.)

### Mid-flight scope-discovery risk

Bundle A's cross-check defers four token-level naming decisions to Bundle A's design.md authoring pass: file-format extension (JSON vs TOML), capability-area name (`provenance-foundation` vs `cflx-trace`), gate-error code name, and the `cflx_trace` MCP-registration question. If the rescope cites any of these by name *before* Bundle A's design.md ratifies them, the rescope is fragile. If the rescope cites them *after*, no fragility. The mid-flight risk is the genuine adversarial concern in the original framing; it cleanly distinguishes Option A (most exposed), Option C (partially exposed during the concurrency window), and Option B (zero exposure).

## Verdict

**Choice**: **Option B with one tightening**: Rescope AFTER Bundle A's *design.md ratifies the deferred identifiers*, not necessarily after Bundle A's full implementation merges.

**Reasoning**: The decisive evidence is cross-check A's four deferred decisions (D4, D5, D7, Q5, Q6) and the per-change scope of `cflx.py validate`. The rescope is a downstream-contract document that names upstream identifiers; rescoping before those identifiers are fixed is structurally fragile (Option A's vice). Rescoping concurrent with their authoring creates two-PR coordination overhead (Option C's vice). Rescoping after their fixation removes both vices. The cost of Option B is documentation-staleness during the window, but that cost is bounded and visible (Phase 9's proposal says deterministic-extraction-only, which is what it currently means; the orchestrator's intent to expand it lives in `cross-check-integrated.md` which is itself a tracked stronghold artefact). Anyone reading Phase 9's proposal during the window can also read the integrated-plan stronghold; the staleness is mitigated by the existence of the cross-check artefacts.

The tightening is: the rescope does not need Bundle A's full apply to merge. It needs Bundle A's *design.md* to be authored and reviewed, because the design.md is where the deferred identifiers get fixed. Once Bundle A's design.md is in (typically committed as part of Bundle A's first or second commit per the cflx-proposal flow), the rescope can author against those final identifiers even if Bundle A's implementation commits are still in flight. This compresses the documentation-staleness window from "Bundle A's full apply timeline" to "Bundle A's design.md authoring timeline", a meaningful reduction (design.md authoring is hours-to-a-day; full apply is days-to-weeks).

This is not a hybrid in the sense of mixing options; it's Option B with the trigger-event tightened from "Bundle A apply lands" to "Bundle A design.md ratifies deferred identifiers". The integrated plan's Wave 4 placement (after Bundle A) is preserved in spirit; the timing is just sharpened to the design.md milestone rather than the apply milestone.

**Confidence**: medium-high. The case-for is grounded in concrete evidence (deferred decisions, per-change validate, cflx workflow primitives). The case-against is the documentation-staleness window, which is real but bounded and partially mitigated by the strongholds. The tightening is not in the integrated plan as written, but it's a refinement, not a contradiction.

**What would flip this**:

1. *Bundle A's design.md absorbs no deferred decisions*: i.e., the cross-check-A deferred questions all resolve token-level identical to what Option A's rescope would have written. In that case the upstream-instability argument evaporates and Option A becomes neutral-vs-Option B; Option A's "compresses cross-phase narrative" advantage then dominates and the verdict flips to Option A.
2. *phase-9.0-tests' proposal-authoring is on the critical path*: if phase-9.0-tests must author against rescoped Phase 9 *before* Bundle A's design.md is in (e.g., because phase-9.0-tests is a hard prereq for some other phase like phase-8-summariser apply), then waiting for Bundle A's design.md is not feasible and the rescope must precede Bundle A entirely. This would flip to Option A.
3. *Bundle A's apply timeline is genuinely short (one commit, one day)*, if Bundle A is small enough that the documentation-staleness window is < 24 hours, Option B's vice disappears, the tightening is unnecessary, and plain Option B (per the integrated plan) is the answer with no caveat.
4. *A new cross-check surfaces a Phase 9 absorption that Bundle A's design needs to know about*, e.g., if the rescope reveals the suggest-engine needs a sidecar field Bundle A hadn't planned, then Bundle A's design.md needs to absorb it, and authoring the rescope first becomes the natural way to surface that requirement. This would flip to Option A.

## What this changes for the integrated plan
